LeetCode Review: Tree vs. Graph

Author: HsingYin Hsieh Date: June 20, 2026  17:09:26 Category: LeetCode Review

Tree

核心概念

tree其實算是一種嚴格受限制的graph

binary tree

  • 核心概念： 就像「上下分明的公司組織架構」。有一個最高領導人（Root，根節點），每個人只能有一個直屬主管（Parent），但可以有兩個下屬（Left/Right Child）。
  • 結構特徵
    1. 絕對沒有環（No Cycles）
    2. 只有一個入口 ( Root )
    3. 節點與邊的關係： 若有 N 個節點，必然剛好有 N-1 條邊

實作結構

  • 通常跟 Linked List 很像，只是 next 變成了 left 和 right

使用情境

  • 資料具備天然的「上下階層」或「主從關係」:如電腦的資料夾結構（DOM Tree）、家族家譜、公司的職位階層。
  • 需要極高效率的搜尋與排序： Binary Search Tree (BST)。在這種樹中，左邊永遠比自己小，右邊永遠比自己大，能讓搜尋、插入的時間複雜度達到完美的 O(log N)。

常考題型

  • 樹的屬性計算： 求樹的最大深度 (Maximum Depth of Binary Tree)、判斷是否平衡。
  • 樹的改造： 翻轉二元樹 (Invert Binary Tree —— 沒錯，就是 Max Howell 去 Google 面試被刷掉的那題）。
  • 二元搜尋樹 (BST) 的特有操作： 利用「左小右大」的特性搜尋或驗證是不是合法 BST。

優缺點

優點

  • 搜尋極快（BST）、結構清晰
  • 因為天生是 Stack 結構，極度適合用遞迴（Recursion）解題。

缺點

  • 退化問題：如果二元樹歪向一邊（變成一條直線），它就會退化成 Linked List，搜尋速度從 O(log N) 掉回 O(N)。

Graph

核心概念

  • 就像「錯綜複雜的社交網路（Facebook 朋友關係）或地圖導航」。節點（Vertex）之間可以亂連，沒有什麼上下階層，誰都可以連到誰。
  • 結構特徵
    1. 可能會有環（Cycles）
    2. 可能有孤島
    3. 沒有固定的 Root (可以從任一個節點開始走訪)

實作結構

  • 電腦沒辦法直接在記憶體儲存「亂連」的結構，所以實務上常用 「鄰接表 (Adjacency List)」，也就是利用你學過的 dict + list/set 來實作。

使用情境

  • 資料關係是「多對多」，且沒有階層： 地圖地鐵路線（節點是車站，邊是軌道）、社交網路（節點是人，邊是好友關係）、網頁連結（網頁互相超連結）。
  • 需要解決「相依性」問題： * 例如 LeetCode 著名的課程安排問題 (Course Schedule)：你想修 B 課程，必須先修 A 課程。這種有方向性的連線叫「有向圖」，可以用拓撲排序 (Topological Sort) 解題。

常考題型

  • 島嶼問題 (經典中的經典)： 給你一個 0 與 1 的矩陣，計算有幾個孤立的 1 的區塊 (Number of Islands)。這本質上就是 Graph 的 BFS/DFS。
  • 相依性與順序問題： 判斷課程能否修完、排課順序 (Course Schedule I & II) $\rightarrow$ 拓撲排序。
  • 最短路徑問題： 網路訊號延遲、迷宮最短步數 $\rightarrow$ BFS（求無權重最短路徑的首選）。

優缺點

優點

  • 表達能力最強，現實世界所有複雜關係都能模擬。

缺點

  • 演算法複雜度較高、容易因為忘記寫 visited 導致 Stack Overflow。空間消耗大。

差異注意

  • 雖然兩者都用 DFS（遞迴）和 BFS（佇列 Queue）來遍歷，但因為結構不同，寫法有一個決定性的差異：
  • Tree 的走訪
    • 因為 Tree 沒有環，你從 Root 開始往下走，絕對不會走回頭路。
    • 代碼特徵： 不需要記錄去過哪裡
  • Graph 的走訪
    • 因為 Graph 有環，如果你不紀錄「哪裡去過」，CPU 就會在環裡面無限繞圈圈（死循環）。
    • 代碼特徵： 必須搭配你熟知的 set() 來當作 visited 記錄。
